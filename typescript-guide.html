JS vs TS

Concept

JavaScript is dynamically typed, meaning types are checked at runtime. TypeScript is a superset of JavaScript that adds static typing, meaning types are checked at compile time (as you write the code). This catches bugs before they even happen.

🟡 JavaScript

You find out a variable is undefined when the user clicks a button and the app crashes.

🔵 TypeScript

Your editor yells at you with a red squiggly line the moment you type the wrong thing.

Comparison Example 
// JavaScript (valid but dangerous)
function add(a, b) { return a + b; }
add(5, "10"); // Returns "510" (string concatenation)

// TypeScript (safe)
function add(a: number, b: number) { return a + b; }
add(5, "10"); // ❌ Error: Argument of type 'string' is not assignable to type 'number'
FeatureJavaScriptTypeScript
TypingDynamic (Checked at runtime)Static (Checked at compile-time)
AutocompleteBasic / GuessedRich, exact, and context-aware
ExecutionRuns directly in the browser/NodeMust be compiled to JS first
Code MaintenanceHarder on large codebasesMuch easier to refactor safely
Custom Data ShapesImplicitExplicit (via interface / type)
When to use which?
🟡 Use JavaScript when...

You are building a quick prototype, a tiny script, or learning the basics of web development. It's fast to set up and requires zero build steps.

🔵 Use TypeScript when...

You are building a production app, working on a team, or dealing with complex data (like this workout tracker). The initial setup pays off by preventing countless bugs.

Interfaces

Core Type

An interface is a contract that says "this object must have exactly these fields with exactly these types." Think of it as a blueprint. TypeScript checks every object you create against the blueprint and warns you if something is wrong.

💡 Plain English
Imagine filling a gym registration form — Name, Weight, Reps. An interface is that form. If you try to submit it with naam instead of name, TypeScript rejects it before the app even runs.
From src/types/index.ts
TypeScript src/types/index.ts 
// Describes a gym exercise object
export interface Exercise {
  id: number;
  name: string;
  category: string;
  isCustom: boolean;
  notes: string;
}

// Describes one logged set — notice Exercise is NESTED inside!
export interface LoggedSet {
  id: number;
  exercise: Exercise;  // another interface used as a type
  weight: number;
  reps: number;
  rpe: number;
  isTopSet: boolean;
}
🔴 Without TypeScript

You could write { nam: 'Bench', wieght: 80 } and only discover the typo when the app crashes at runtime.

🟢 With Interface

TypeScript highlights the error immediately — a red underline appears in your editor before you even save.

Type Aliases

Core Type

A type alias gives a name to any type including primitive combinations and unions. Uses the keyword type instead of interface.

TypeScript src/types/index.ts 
// A type alias — gives a name to a union of string literals
export type StrengthGrade = 'PLATINUM' | 'GOLD' | 'SILVER' | 'BASELINE';
Featureinterfacetype
Object shapes✅ Perfect for this✅ Works too
Union types❌ Can't do this✅ Primary use case
Extends✅ extends keyword✅ Intersection &
Re-openable✅ Can merge❌ Cannot
Rule of Thumb for This Project
Use interface for objects (like Exercise, LoggedSet). Use type for unions of specific string values (like StrengthGrade).
03

Union Types

Core Type

A union type says "this value can be ONE of these types." Written with a | pipe between each option.

TypeScript src/types/index.ts 
// 1. Union of STRING LITERALS — only these four words allowed
type StrengthGrade = 'PLATINUM' | 'GOLD' | 'SILVER' | 'BASELINE';

// Writing 'DIAMOND' would cause an error! ❌
const grade: StrengthGrade = 'DIAMOND';
// Error: Type '"DIAMOND"' is not assignable to type 'StrengthGrade'

// 2. Union with PRIMITIVES — value or null
rir: number | null;

// 3. Union status literals (from NextSessionTarget)
status: 'PENDING' | 'COMPLETED' | 'SKIPPED';
You type 'DIAMOND'
→
TS checks union
→
❌ Error before runtime

Optional & Nullable Fields

Core Type

Not every field is always present. TypeScript gives you two ways to express this:

SyntaxMeaningExample from project
field?: TypeProperty may be missing entirelyexerciseId?: number
field: Type | nullAlways present but value may be nullrir: number | null
field?: Type | nullMay be absent OR nullrpe?: number | null
TypeScript src/types/index.ts — SetLogRequest 
export interface SetLogRequest {
  exerciseId?: number;        // optional — may be missing
  exerciseName?: string;      // optional — may be missing
  sessionId: number;          // required — must always be there
  weight: number;             // required
  rpe?: number | null;        // optional AND nullable
  rir?: number | null;        // optional AND nullable
  setNote?: string | null;   // optional AND nullable
  isTopSet: boolean;          // required
}
💡 Why both optional AND nullable?
rpe?: number | null means the user might not have submitted RPE (key missing), OR they cleared it to null. The backend can distinguish these two!
05

Generics

Advanced

Generics let you write reusable code that works with different types while staying type-safe. Think of <T> as a type variable — a placeholder that gets filled when you use it.

💡 Analogy
A lunch box is a container. Box<Rice> or Box<Pasta> — the shape is the same but you always know what's inside.
TypeScript src/context/AuthContext.tsx 
// useState<T> — T is the generic type slot
const [token, setToken] = useState<string | null>(null);
//                            ↑ T = "string | null"
//  Calling setToken(42) would now be an error!

const [user, setUser] = useState<User | null>(null);
//                            ↑ T = "User | null"

// createContext<T> — tells React what the context value looks like
const AuthContext = createContext<AuthContextType | undefined>(undefined);
//                              ↑ Starts as undefined, becomes AuthContextType once Provider wraps app
Write useState<T>
→
TypeScript fills in T
→
All usages are checked
→
✅ Safe code

React.FC & Typed Props

React

React.FC = React Functional Component. The generic parameter <{ ... }> declares what props the component accepts.

TypeScript src/components/ProtectedRoute.tsx 
// React.FC<Props> — typed functional component
const ProtectedRoute: React.FC<{ children: React.ReactNode }> = ({ children }) => {
//                  ↑ Generic says: accepts one prop called `children`
//                    which must be React.ReactNode (any valid JSX)
  const { token } = useAuth();
  if (!token) return <Navigate to="/login" replace />;
  return <>{children}</>;
};

// AuthProvider — same pattern, children comes from ReactNode import
export const AuthProvider: React.FC<{ children: ReactNode }> = ({ children }) => {
  // ...state and functions
  return (
    <AuthContext.Provider value={{ token, user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
};
07

Context API with TypeScript

React

React Context lets you share state globally without prop drilling. TypeScript makes it fully type-safe via interfaces.

TypeScript src/context/AuthContext.tsx 
// STEP 1: Define the shape
interface AuthContextType {
  token:   string | null;
  user:    User | null;
  login:   (token: string, username: string) => void;  // function type!
  logout:  () => void;
}

// STEP 2: Create context — starts undefined, filled by Provider
const AuthContext = createContext<AuthContextType | undefined>(undefined);

// STEP 3: Wrap your app — now all children have access
<AuthProvider>
  <AppContent />    // can call useAuth() anywhere in here
</AuthProvider>
📍 In App.tsx

<AuthProvider> wraps the whole app, making auth data available everywhere inside.

🪝 In any component

Call useAuth() to get token, user, login, logout — all fully typed!

08

Custom Hooks with TypeScript

React

A custom hook is a function starting with use that can call other React hooks. TypeScript defines exactly what it returns.

TypeScript src/context/AuthContext.tsx 
// useAuth — custom hook returning the auth context
export const useAuth = () => {
  const context = useContext(AuthContext);

  // Type guard — ensures hook is used inside AuthProvider
  if (context === undefined) {
    throw new Error('useAuth must be used within an AuthProvider');
  }
  // After the throw, TS KNOWS context is AuthContextType (not undefined)
  // This is called TYPE NARROWING

  return context;
};

// Usage — token and logout are fully typed
const { token, logout } = useAuth();
//     ↑ string | null       ↑ () => void
🔑 Type Narrowing
After the if (context === undefined) throw, TypeScript narrows the type — it now knows context is AuthContextType (not undefined). The error also helps you debug fast in development.

Record<K, V> Type

Advanced

Record<Keys, Values> creates an object type where every key is type K and every value is type V. It ensures you handle every possible key — no case forgotten.

TypeScript src/components/GradeBadge.tsx 
import type { StrengthGrade } from '../types';

// Record<StrengthGrade, { bg, text, glow }>
//  Every grade MUST have a style entry.
//  Adding 'DIAMOND' to StrengthGrade but forgetting here = ERROR!
const gradeStyles: Record<StrengthGrade, { bg: string; text: string; glow: string }> = {
  PLATINUM: { bg: 'linear-gradient(135deg,#00f5a0,#00d9f5)', text: '#08090d', glow: '...' },
  GOLD:     { bg: 'linear-gradient(135deg,#ffd166,#ffab00)', text: '#08090d', glow: '...' },
  SILVER:   { bg: 'linear-gradient(135deg,#a0aec0,#718096)', text: '#08090d', glow: '...' },
  BASELINE: { bg: 'var(--color-surface-raised)', text: '...', glow: 'none' },
};

// Usage — always type-safe lookup
const style = gradeStyles[grade];
⚠️ Without Record
Using const gradeStyles: any = {...} loses all safety. A new grade could be added to the union and the style mapping would be silently missing — until the UI breaks in production.

Utility Types

Built-in TS

TypeScript ships with built-in utility types — helper generics that transform existing types.

Utility TypeWhat it doesWhere used in project
Partial<T>Makes every field of T optionalcreateExercise(ex: Partial<Exercise>)
Record<K,V>Object type with keys K and values VgradeStyles: Record<StrengthGrade,{...}>

TypeScript src/api/api.ts 
// Partial<Exercise> = { id?: number; name?: string; ... }
//   All fields become optional — perfect for CREATE requests
//   where you don't have an id yet!
export const createExercise = (ex: Partial<Exercise>) =>
  API.post<Exercise>('/exercises', ex).then(r => r.data);

// Record<string, unknown> — safer than any for a flexible patch body
export const patchSession = (id: number, data: Record<string, unknown>) =>
  API.put<void>(`/sessions/${id}`, data);

Axios Generics

Advanced

Axios's HTTP methods accept a generic type that tells TypeScript what the response data shape will be.

TypeScript src/api/api.ts 
// API.get<Exercise[]> → response.data is typed as Exercise[]
export const getExercises = () =>
  API.get<Exercise[]>('/exercises').then(r => r.data);

// post<SetResponseDTO> — sending SetLogRequest, getting SetResponseDTO back
export const logSet = (request: SetLogRequest) =>
  API.post<SetResponseDTO>('/sets', request).then(r => r.data);

// put<void> — PUT that returns nothing meaningful
export const patchSession = (id: number, data: Record<string, unknown>) =>
  API.put<void>(`/sessions/${id}`, data);

// Interceptors — automatically attach the auth token to every request
API.interceptors.request.use((config) => {
  const token = localStorage.getItem('iron_ledger_token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});
🎯 Autocomplete

After await getExercises(), TypeScript knows you have Exercise[] — every field has autocomplete in your editor.

🛡️ Documentation

The generics act as living documentation — you can see exactly what the API returns just by reading the function signature.

Typed Arrow Functions

Pattern

Arrow functions in TypeScript can have typed parameters and return types. Several patterns appear in this project:

TypeScript Various files 
// 1. Function typed in an interface
login: (token: string, username: string) => void;

// 2. Implementation matching the type
const login = (newToken: string, username: string) => {
  localStorage.setItem('iron_ledger_token', newToken);
  setToken(newToken);
};

// 3. Typed callback parameters — from SessionAccordionCard.tsx
session.exercises?.map((block: any, i: number) => (/* JSX */))

// 4. Utility function with explicit return type — from SessionAccordionCard.tsx
function calculateSimpleE1RM(weight: number, reps: number): string {
  if (reps === 1) return weight.toFixed(1);
  if (reps >= 2 && reps <= 5) return (weight * 36.0 / (37.0 - reps)).toFixed(1);
  return '—';
}
💡 import type
You'll see import type { Exercise } from '../types' throughout the project. The type keyword means this import is erased completely from compiled JavaScript — it exists only for type checking. Slightly smaller bundle, no runtime cost.
🔢 void

Return type void = "this function doesn't return a useful value." Like logout() — it does work but returns nothing.

🏷️ any

any turns off TypeScript for that value. Used in a few places (session: any) when the full type is complex. A trade-off.

❓ Optional chaining ?.

session.exercises?.map(...) — if exercises is null/undefined, returns undefined instead of throwing.
